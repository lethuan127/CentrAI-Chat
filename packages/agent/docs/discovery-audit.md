## 1. External consumers (this repo)

Only **`apps/api`** depends on `@centrai/agent`. Imports today:

| Symbol | File |
| ------ | ---- |
| `createCentrAiChatStream` | `chat/chat.controller.ts` |
| `buildSystemPrompt` | `provider/provider.service.ts` |

`createCentrAiPostgresStore` / `PostgresStore` are **not** imported by `apps/api` until you add a provider and pass `postgresStore` into `createCentrAiChatStream`.

`apps/worker` does **not** import this package yet.

## 4. Optional cleanups (breaking for hypothetical external users only)

1. **`tools/invocation-router.ts`** — Remove module and stop exporting `ToolInvocationCall` / `ToolInvocationHandler` until the router exists; reintroduce with the first `invokeTool` implementation.
2. **`RequestContext`** — Remove `compile/request-context.ts` and export if you do not want forward-declared types in the package.
3. **`CENTRAI_MASTRA_PG_STORE_ID`** — Stop re-exporting from `index.ts` if callers should only use `createCentrAiPostgresStore`.
4. **`mastraAgentRuntimeAdapter`** — Stop re-exporting if every caller should go through `createRuntimeAdapter` (tests can import from `./mastra/adapter.js` inside the package only).

## 6. API alignment gap (not “dead,” but inconsistent)

`ProviderService.buildInstructions` duplicates some of what `runtimeAgentDefinitionFromPersisted` + `systemPromptFromDefinition` would produce from a DB agent row. Consolidating later would reduce drift but is a separate refactor.
